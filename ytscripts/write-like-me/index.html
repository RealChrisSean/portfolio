Write Like Me, Not For Me

Full script for the Speak It Chrome extension video. Runtime: ~8-9 min

~8-9 min runtime

[HOOK]

"I didn't want AI to write FOR me. I wanted it to write LIKE me."

"See, I started using ChatGPT to clean up my messages. Fix grammar, spelling, all that. And it works. Except now everything I write sounds like this:"

(Read in a mocking corporate tone)

"'I hope this email finds you well. I wanted to circle back regarding our previous discussion.'"

(Back to normal)

"That's not me. That's ChatGPT. Nobody talks like that. I don't talk like that."

"So I built something. Let me show you."

[SHOW THE PAYOFF]

(Screen recording: Chrome open with Gmail, Slack, X in tabs)

"I'm going to say the same thing on three different platforms. Watch what happens."

(Click mic button. Speak:)

"Just finished the new feature and it's working way better than I expected"

(Show Gmail output:)

"Gmail. It added a greeting, fixed the capitalization, added a sign-off. Makes sense for email."

(Switch to Slack. Same sentence:)

"Slack. Casual. Lowercase. No sign-off. Because that's how people actually talk on Slack."

(Switch to X. Same sentence:)

"X. Kept it short. No extra formatting."

"Same words I spoke. Different formatting based on where I'm typing. And it learns how I write over time. Without storing my messages."

"That last part is the tricky bit. Let me explain."

[THE REAL PROBLEM]

"Most apps that learn your style keep everything you write. Every Slack message. Every email. Every tweet. Sitting in a database somewhere. Maybe encrypted, maybe not. Either way, your words are stored somewhere you don't control."

"I didn't want that. Not for myself. And definitely not if anyone else was going to use this."

"So I asked a weird question: can you teach AI how someone writes without showing it what they write?"

"Turns out you can. But getting there broke me a little."

[THE STRUGGLE]

"Here's the thing about building a Chrome extension that types into other apps. You'd think there's a universal way to insert text into a webpage. There isn't."

"Gmail, Slack, X, Notion. They all handle text input differently. Something called contentEditable. It's how web apps let you type rich text. Bold, italics, links, all that stuff. Every app implements it their own way."

"Gmail? I call the insert function, text goes in. Done."

"X? You have to delete the existing content first or it breaks. Weird, but okay."

"Notion?"

(Beat. Look at camera.)

"The text came out vertical. Each letter stacked on top of the next. Like a totem pole."

(Pause)

"I'm not joking. I called the same function that worked everywhere else, and Notion rendered it vertically. The fix? There is no fix. You copy to clipboard and tell the user to paste."

(Show the code:)

navigator.clipboard.writeText(text);
showMessage('Press Cmd+V to paste');

"That's the solution for Notion. Clipboard and a prayer."

"And that's before I got to audio streaming. I had 10 lines of code that grab audio from your mic and send it to Deepgram for transcription."

(Show the code:)

processor.onaudioprocess = (event) => {
  const inputData = event.inputBuffer.getChannelData(0);
  const pcmData = new Int16Array(inputData.length);
  for (let i = 0; i < inputData.length; i++) {
    const sample = Math.max(-1, Math.min(1, inputData[i]));
    pcmData[i] = sample < 0 ? sample * 0x8000 : sample * 0x7fff;
  }
  socket.send(pcmData.buffer);
};

"10 lines. Took me two days. The sample rate has to match exactly. The buffer size matters. And if you get that Int16 conversion wrong, Deepgram hears static. No error. No warning. Just static."

"Building this was a lot of Googling, guessing, and asking myself why I started this project."

"But eventually I figured it out."

[THE BREAKTHROUGH]

"So here's how it actually works."

"Instead of storing your messages, I store statistics. Sentence length. Formality score. Whether you use contractions. Your common phrases. Your typical greetings. Your sign-offs."

(Show schema on screen:)

CREATE TABLE user_style_profiles (
  user_id VARCHAR(255) PRIMARY KEY,
  avg_sentence_length FLOAT,
  formality_score FLOAT,
  uses_contractions BOOLEAN,
  top_phrases JSON,
  greetings JSON,
  signoffs JSON
);

"Look at this table. Notice what's NOT here."

(Beat)

"No message content column. I store HOW you write, not WHAT you write."

"I call it a style fingerprint. Not a transcript."

"When you speak, the app transcribes it, detects where you're typing, pulls your style profile, and sends it to Claude with instructions."

(Show the prompt code:)

return `Format this transcript for ${context}.

User's writing style:
- Tone: ${formality}
- Average sentence length: ~${profile.avg_sentence_length} words
- ${profile.uses_contractions ? "Use contractions." : "Avoid contractions."}
- Preferred greetings: ${profile.greetings.join(", ")}
- Preferred sign-offs: ${profile.signoffs.join(", ")}

Rules:
1. ONLY add punctuation and paragraph breaks
2. Remove filler words: um, uh, like, basically
3. Keep EVERY other word exactly as spoken
4. Do NOT rewrite or rephrase their language`;

"Claude gets constraints, not content. 'This person writes casually, short sentences, uses contractions, starts with Hey, ends with Thanks.'"

"Those rules at the bottom are critical. Without them, Claude will 'improve' your words. Make things clearer. More professional. More ChatGPT. But I don't want my voice replaced. I want it cleaned up. There's a difference."

"Your actual words? Gone as soon as the request finishes. Never stored."

[THE DATABASE PROBLEM]

"Now I had a new problem. Where do I store all this?"

"I needed two different types of data."

"One: User style profiles."

"Relational data. User ID, formality score, sentence length, greetings, sign-offs."

(Show schema)

"Standard SQL stuff."

"Two: Vector embeddings."

"For finding users with similar writing styles. Your style stats get converted to an embedding. 1536 dimensions."

(Show schema:)

CREATE TABLE style_embeddings (
  user_id VARCHAR(255) PRIMARY KEY,
  embedding VECTOR(1536),
  VECTOR INDEX idx_embedding (embedding)
);

"The standard advice? Postgres for relational. Pinecone for vectors. Two databases. Two connections. Two bills. Two things that can break."

"For a side project? Nah."

[HOW TIDB HANDLES BOTH]

"TiDB does both. One database."

(Show vector query:)

SELECT user_id, VEC_COSINE_DISTANCE(embedding, ?) AS similarity
FROM style_embeddings
ORDER BY similarity
LIMIT 5;

"Find users with similar writing styles. Cosine distance. Same database that stores the relational data."

"I work for PingCAP so take that with a grain of salt. But I chose TiDB because it solved my problem. Relational and vectors without managing two separate systems."

[CLOSE]

"Here's what I learned building this."

"Personalization doesn't require surveillance. You can teach AI your patterns without teaching it your secrets. Statistics are enough. Sentence length. Formality. Phrases. A fingerprint."

"I didn't want AI to write FOR me. I wanted it to write LIKE me."

"That's what I built."

"Code's on GitHub if you want to look at it. Link's in the description."
